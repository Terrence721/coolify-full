## Done

### De-commercialization

- Removed the "Love Coolify? Support our work" sponsorship popup and its Settings toggle.
- Removed the "Sponsor us" navbar link.
- Trimmed the navbar's bottom menu down to just Theme and Logout (removed What's New, Upgrade, Feedback).
- Fully removed the Stripe/subscription billing subsystem: Stripe actions, webhook handler, subscription pages, the `subscriptions` database table, and every code path that gated features on subscription/payment status (scheduled jobs, server cleanup, admin tools, notifications). Server-count limits still exist as a plain team setting (no billing tie-in) but are effectively unlimited for a self-hosted instance.
- Removed the `stripe/stripe-php` Composer dependency.
- Removed two Hetzner Cloud affiliate-link blocks ("Coolify's affiliate link... supports us (€10) and gives you €20") — one in the shared `Security\CloudProviderTokenForm` Livewire component, one already carried over into the converted `Security\CloudTokens.jsx` page.

> **Milestone (2026-07-12):** the dev environment **moved off Windows entirely, into a WSL2 Ubuntu distro's native filesystem** — the repo now lives at `/root/projects/coolify-full` inside WSL2, not under `C:\Users\...`. Reason: `docker exec coolify-vite yarn build` was taking **over 3 hours** to complete. Root-caused via `/proc/<pid>/stat`/`wchan` to `p9_client_rpc` — every file the build touched was crossing the 9P protocol bridge Docker Desktop's WSL2 backend uses to reach the Windows NTFS host filesystem, with Windows Defender re-scanning each crossing on top; the same build run natively on the Windows host (bypassing the bridge) took under 10 seconds, proving the bridge itself was the bottleneck, not the build. Moving the repo into WSL2's own ext4 filesystem means both `coolify` and `coolify-vite`'s bind mounts read/write natively with no bridge left to cross — not just for builds, for every container operation. Result: that same `yarn build` now completes in **~2 seconds**, and the full Pest suite dropped from ~150–170s to **~31s**; enabling PHP OPcache in dev the same day added a further per-request win on top. Fixed to `$proxy->set('status', ...); $this->server->save();`, matching every other of the 9 existing call sites of `$server->proxy->set(...)` in the codebase (`CheckProxy`, `StartProxy`, `CreateHetznerServer`, `ServerController`, `ServerProxyController`'s API controller, `HasProxyConfiguration`, `ProxyStatusChangedNotification`). (2) `CheckAndStartSentinelJob::handle()` called `json_decode($sentinelFound, true)` where `$sentinelFound` comes from `instant_remote_process_with_timeout(['docker inspect coolify-sentinel'], $this->server, false, 10)` — declared `?string` and called with `$throwError = false`, so it returns `null` (not a thrown exception) whenever the container doesn't exist (e.g. a freshly provisioned server that's never run Sentinel). `json_decode()`'s first parameter is non-nullable under `strict_types=1`, so this crashed with a `TypeError` on exactly the case the surrounding code was clearly trying to handle gracefully (`data_get(..., 'exited')`'s fallback never got a chance to run). Fixed with `json_decode($sentinelFound ?? '', true)`. (3) The recurring `mb_split()` "`Call to undefined function Illuminate\Support\mb_split()`" errors on `artisan list` were investigated and confirmed **not** a regression of the 2026-07-15 fix directly above (`EventServiceProvider` still correctly has `shouldDiscoverEvents(): false`) — instead traced to a stray host PHP 8.5 CLI binary invoking `artisan list` outside the dev container (`function_exists('mb_split')` is `false` on the host's PHP 8.5.4, `true` inside the `coolify` container's PHP 8.4.23), i.e. exactly the pre-existing, already-documented "editor-extension detection only" quirk from this file's Development Environment section — no application code to fix. New regression tests: `tests/Unit/ServerCheckJob/UpdatesProxyStatusTest.php` and `tests/Unit/CheckAndStartSentinelJob/HandlesMissingContainerTest.php` (the latter needed a new `App\Jobs`-namespaced override of the global `instant_remote_process_with_timeout()` helper, `tests/Support/Fakes/jobs_remote_process_overrides.php` + `JobsRemoteProcessFake`, following the existing per-namespace helper-override pattern used for `App\Actions\Database`/`App\Actions\Application`). Both new tests confirmed failing against the pre-fix code with the exact reported error message before the fix landed. Pint clean, PHPStan 0 new errors, full suite 1201 passed (1199 + 2 new).

- **2026-07-19 — real `CleanupStaleMultiplexedConnections` crash fixed, found from a live Horizon log paste.** The user pasted a live `TypeError` from a scheduled job: `substr(): Argument #1 ($string) must be of type string, null given` at `app/Jobs/CleanupStaleMultiplexedConnections.php:176`. `cleanupStaleConnections()` lists SSH mux socket files via `Storage::disk('ssh-mux')->files()`, then for each one still passing its `ssh -O check` liveness probe, reads the socket file's content via `Storage::disk('ssh-mux')->get($muxFile)` to parse an "established at" timestamp out of it with `substr($muxContent, 37)`. `Storage::get()` is genuinely nullable — Laravel's `FilesystemAdapter::get()` catches `UnableToReadFile` and returns `null` rather than throwing (confirmed by reading the framework source directly, not assumed) whenever the file can't be read, which happens for real here: a mux socket file listed by `files()` can be torn down by a concurrent SSH connection closing before this same loop iteration reaches `get()` on it — a genuine TOCTOU race, not a hypothetical. Passing that `null` into `substr()` fatally errors under this codebase's `strict_types=1`, and — critically — since nothing in `cleanupStaleConnections()`'s loop or `handle()` itself catches it, the exception aborts the *entire* job: every remaining mux file in that same loop, plus all 3 cleanup steps that run after it (`cleanupNonExistentServerConnections`, `cleanupOrphanedSshProcesses`, `cleanupOrphanedCloudflaredProcesses`), silently never ran whenever this race fired. Fixed by adding a null check that calls the file's own already-established `removeMultiplexFile($muxFile, 'content_unreadable')` cleanup path and `continue`s — the same pattern the two sibling checks in the same block (`server_not_found`, `connection_check_failed`) already used, just missing for this third failure mode. TDD'd first: `tests/Unit/Jobs/CleanupStaleMultiplexedConnectionsTest.php` mocks the `ssh-mux` disk directly (`Mockery::mock()` returning `[$muxFile]` from `files()` and `null` from `get($muxFile)`, since reproducing a genuine filesystem race deterministically isn't practical) and fakes `Process` so the `ssh -O check` probe reports success, reaching the buggy line; confirmed the test fails with the exact reported `TypeError` against the reverted code before restoring the fix and confirming it passes. `tests/` isn't in `phpstan.neon.dist`'s analyzed `paths:`, so the 2 Mockery-typing-related findings PHPStan reports when the test file is explicitly named don't apply to the real baseline (same out-of-scope situation as `app/Helpers/`, confirmed in Phase 52/56) — the actual `app/Jobs/` fix itself is fully clean. Not folded into the numbered PHPStan-phase sequence since it has zero baseline delta (`[OK] No errors` before and after) — this was found by reading a live crash log, not by a phase's own investigation. Pint clean, full suite 1212 passed (1211 + 1 new).
