Had worked locally only because the empty directory tree was still sitting on disk from before the deletions. Pure CI-config fix, no baseline change (1572 → 1572).
- Phase 5 (`e106944d1`) — Extract the storages() family into a shared ManagesApiResourceStorages trait. Applied the existing `app/Http/Controllers/Concerns/` extraction pattern (same loose-typing + `instanceof` branching style as `ManagesResourceStorages`, no abstract hook methods) to `ApplicationsController`, `DatabasesController`, and `ServicesController`'s `storages()`/`create_storage()`/`update_storage()`/`delete_storage()` family — mechanical for Application/Database, with a real `instanceof Service` branch for the aggregate-across-children shape. Verified against the storages Tier A characterization suite (49 tests, unchanged) after each controller's extraction. Baseline actually *grew* 1570 → 1606 (+36): the new trait's necessarily loose `Model $resource` typing (Application/StandaloneDatabaseInstance/Service share no common interface exposing `persistentStorages`/`fileStorages`) added entries — an accepted tradeoff, the same class of pre-existing limitation as the already-baselined `StandaloneDatabaseInstance` entries.
- Phase 6 (`ac32869a0`) — Chip away at the PHPStan baseline: add return/param types to shared traits. PHPStan attributes a trait method's missing-type errors once per consuming class, so an untyped method on a widely-shared trait was showing up as dozens of duplicate baseline entries. Added explicit return types (and, for `HasSafeStringAttribute`, parameter types) to `HasStandaloneDatabaseCommon` (used by all 8 database engines), `HasSafeStringAttribute` (18 consumers), `ClearsGlobalSearchCache`, `HasResourceStatus`, `HasResourceCleanup`, and `HasResourceLinks`, plus the small number of per-model overrides needing matching signatures (`Environment`/`Tag::customizeName()`, `ServiceApplication`/`ServiceDatabase::isBackupSolutionAvailable()`). Two real (if harmless) things surfaced only once the trait methods were properly typed: a dead `?? collect()` in `deleteVolumes()` removed (`MorphMany::get()` never returns null), and one `environment_variables()->get('value')` call switched to `->get(['value'])` to match the relation's actual parameter type (identical runtime behavior either way). Baseline: 1609 → 1323 — the largest single-phase drop of the whole initiative, since a trait fix collapses N duplicate per-consumer entries into one. Pint clean, PHPStan 0 errors, full suite 1190/1190 passed.
- Phase 7 (`8a5b62890`) — Chip away at the PHPStan baseline: type the metrics and env-analyzer traits. `HasMetrics` (Server + all 8 database engines) and `EnvironmentVariableAnalyzer` (`ManagesResourceEnvironmentVariables`, `ApplicationDeploymentJob`, and both `Project*ConfigurationController` classes) were missing array-shape PHPDoc on otherwise already-typed methods. Added precise shapes (the metrics tuple `array<int, array{0: int, 1: float}>`, the build-warning `array{variable, value, affects, issue, recommendation}`) instead of bare `array`/`mixed`. Baseline: 1323 → 1257. Pint clean, PHPStan 0 errors, full suite 1190/1190 passed.
- Phase 8 (`4fa0bba73`) — Chip away at the PHPStan baseline: fix camelCase Attribute-accessor property access. Several `Attribute`-cast accessors (`Server::getIp()`, `Application::gitWebhook()`/`gitBranchLocation()`/`gitCommits()`, `Service::isDeployable()`) were being read via their camelCase method name (e.g. `$server->getIp`) at 13 call sites. This worked at runtime purely because `Str::camel()` is idempotent on an already-camelCase key, but it's not the form Eloquent/Larastan actually documents (each model's own `@property-read` docblock already lists the snake_case form, e.g. `$get_ip`) — so every call site was invisible to static analysis and had to be baselined. Switched all 13 to the documented snake_case property (`get_ip`, `git_webhook`, `git_branch_location`, `git_commits`, `is_deployable`); identical runtime behavior, now statically checked. > **Milestone (2026-07-12):** the dev environment **moved off Windows entirely, into a WSL2 Ubuntu distro's native filesystem** — the repo now lives at `/root/projects/coolify-full` inside WSL2, not under `C:\Users\...`. Reason: `docker exec coolify-vite yarn build` was taking **over 3 hours** to complete. Root-caused via `/proc/<pid>/stat`/`wchan` to `p9_client_rpc` — every file the build touched was crossing the 9P protocol bridge Docker Desktop's WSL2 backend uses to reach the Windows NTFS host filesystem, with Windows Defender re-scanning each crossing on top; the same build run natively on the Windows host (bypassing the bridge) took under 10 seconds, proving the bridge itself was the bottleneck, not the build. Moving the repo into WSL2's own ext4 filesystem means both `coolify` and `coolify-vite`'s bind mounts read/write natively with no bridge left to cross — not just for builds, for every container operation. Result: that same `yarn build` now completes in **~2 seconds**, and the full Pest suite dropped from ~150–170s to **~31s**; enabling PHP OPcache in dev the same day added a further per-request win on top. See `docs/command.md`'s "RESOLVED" section for the full root-cause writeup and the "WSL2 migration" section for setup steps, and Section 105 of the migration doc for how this surfaced during Phase 50's work.

The Inertia frontend and `/api/v1/*` both run through Laravel's own standard `web`/`api` middleware groups (plain session + CSRF-token-value matching), which don't care about Origin/Host at all. Confirmed directly: `curl http://127.0.0.1:8080/login -H "Host: 192.168.1.50:8000"` → `HTTP 200`, real page HTML, no rejection. Not a blocker.
  3. **CORS** (`config/cors.php`) — `allowed_origins => ['*']`, already permissive; not the security boundary for a same-origin Inertia SPA in the first place. Not a blocker.
  4. **The actual, real blocker (dev-only): Vite's dev server asset URLs.** `vite.config.js`'s `server.origin` is set from `VITE_HOST`, which `docker-compose.dev.yml` defaults to the literal string `"localhost"` (not empty/unset) — so Vite hardcodes `http://localhost:5173` into every asset tag regardless of what Host header the actual request used. Confirmed with real output: `curl .../login -H "Host: 192.168.1.50:8000"` returns HTML containing `<script src="http://localhost:5173/resources/js/app.js">` verbatim — a LAN device receives working page HTML (per point 2) but its own browser then tries to fetch JS/CSS from *itself* on port 5173, not the dev machine, silently failing every asset load. Fixable by setting `VITE_HOST=<host-machine-LAN-IP>` in `.env` for dev sessions that need cross-device testing — `vite.config.js` already supports this via env var, just isn't documented as a cross-device requirement anywhere. **Production is unaffected**: `yarn build` serves pre-built static assets from the same origin as the app itself, no separate Vite dev port involved.

  Added a checklist item to `docs/smoketest.md`'s Setup section documenting exactly this (the LAN-IP visit, the expected dev-only asset-loading symptom, the `VITE_HOST` fix, and the production non-issue) so it's a standing regression check, not a one-time investigation.

'none')`, matching the existing `match` arm that already maps `'none'` to a null encryption.
